# 说明文档

## 一、项目规划

### 1.1 背景与目标
本项目用于对代码/配置/提示词等进行安全扫描。新增需求：引入大模型能力，增强对 skill（含代码、配置、提示词与依赖等）进行安全扫描分析，能够更多识别潜在“有害点”，并给出可落地的修复建议。

### 1.2 交付范围（严格限定）
1. 大模型 Provider 抽象与配置模型（默认关闭；API Key 仅从环境变量读取）。
2. LLM 发现能力：在无规则命中的情况下，发现可疑/有害风险点并结构化输出。
3. LLM 增强能力：对现有规则/AST 命中进行语义复核、误报压制、分级与修复建议补全。
4. 报告输出增强（md/json/sarif）与阈值统计联动，并补齐测试用例与离线回放夹具。

### 1.3 非范围（本轮不做）
1. 规则库大规模重写或全新规则体系设计。
2. 引入复杂的 UI 平台或服务端系统（保持 CLI 工具形态）。
3. 对所有语言做深度数据流分析（仅在现有基础上做增强与发现）。

### 1.4 计划时间节点（不可延期）
- 2026-02-05：方案确认与代码库摸底完成；创建说明文档；开始开发。
- 2026-02-06：完成 Provider/配置模型、LLM 发现能力与 Runner 串接。
- 2026-02-07：完成 LLM 增强能力、报告输出与阈值统计；补齐测试与离线回放；完成验收。

## 二、实施方案

### 2.1 总体架构
采用“两段式 + 发现”混合策略：
1. 确定性扫描：现有 RegexScanner/AstScanner 产出候选问题点。
2. LLM 发现：基于启发式筛选与片段裁剪，识别“无规则命中”的有害点。
3. LLM 增强：对候选问题点做语义归因、风险分级、置信度与修复建议补全。

### 2.2 输入控制与安全合规
1. 默认关闭外部 LLM 调用；启用需显式配置 provider/model/baseUrl，并通过环境变量提供 API Key。
2. 输入最小化：仅发送命中片段 ±N 行上下文与必要配置片段。
3. 脱敏默认开启：对疑似 Token/Key/私钥等进行替换后再发送。
4. 失败降级：LLM 失败不影响基础扫描流程，确保 CI 可用。

### 2.3 输出与可审计性
1. LLM 输出强结构化 JSON（固定 schema），解析失败也会产出可追踪的“需人工复核”记录。
2. 报告包含模型信息、提示词摘要（hash）与统计信息，便于回溯与审计。
